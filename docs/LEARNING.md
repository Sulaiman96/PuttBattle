# LEARNING.md — Unreal concepts glossary (append-only)

Maintained by agents per CONVENTIONS §12: the first time a task uses an Unreal concept, add a 1–3 line entry here and reference it in the task summary instead of re-explaining. Audience: a senior C#/.NET engineer new to Unreal. C# analogies in italics where they genuinely fit.

- **UObject & garbage collection** — nearly everything in UE derives from `UObject` and is garbage-collected, but only pointers marked `UPROPERTY()` count as references; a raw `UObject*` member can dangle after GC. *CLR GC, but reachability is opt-in via attributes.*
- **UPROPERTY / UFUNCTION / UCLASS** — macros parsed by Unreal Header Tool to generate reflection code; they gate GC tracking, editor exposure, Blueprint access, replication, and serialization. *Attributes + source generators — but mandatory for the engine to "see" your members at all.*
- **Module** — UE's buildable unit, each with a `Build.cs` declaring dependencies. Our game is one module: `PuttBattle`. *A csproj: `Build.cs` ≈ the project file, `PublicDependencyModuleNames` ≈ PackageReference.*
- **Actor / Pawn / Component** — Actor: anything that can exist in a level. Pawn: an Actor a controller can possess (our ball). Components: composable behaviours attached to Actors (our ShotComponent). *Formalised composition-over-inheritance.*
- **GameMode / GameState / PlayerState / PlayerController** — the framework quartet. GameMode: server-only rules referee. GameState: replicated facts everyone sees (timer, phase). PlayerState: replicated per-player facts (strokes, score). PlayerController: one player's input and will. All multiplayer design hangs off this split — internalise it early.
- **Replication** — server-authoritative state sync. `UPROPERTY(Replicated)` variables flow server→clients; RPCs (`Server_`/`Client_`/`Multicast_`) are cross-machine calls. Clients never decide truth: they *request* (RPC) and *render* (`OnRep_`).
- **DataAsset** — an editor-authored object of designer data referenced by code. Our whole content model (powerups, surfaces, holes, scoring) is DataAssets. *Strongly-typed appsettings objects, edited in a GUI.*
- **Blueprint** — visual scripting and the editor-side subclassing system. Project rule: C++ owns logic; Blueprint subclasses exist only to wire assets (meshes, curves, sounds) onto C++ bases.
- **GAS (Gameplay Ability System)** — Epic's heavyweight plugin for ability kits: AbilitySystemComponent, AttributeSets, GameplayEffects with stacking/magnitude curves, client-side ability prediction, animation integration. Industry-standard for RPGs/hero shooters; famously steep to learn. *Roughly: a full MediatR-style framework plus an effects engine, when our game needs four well-named structs.* We deliberately don't use it (D22) — but we stole its good ideas: GameplayTags, data-driven definitions, one central attribute struct, effect lifetime management. Note: **GameplayTags are a separate engine module, not part of GAS** — using tags ≠ using GAS.
- **Collision object channels & profiles** — collision is configured in `DefaultEngine.ini`, not code. *Object channels* (`PB_Ball/PB_Floor/PB_Wall`) classify what a primitive *is*; a *profile* (`PB_BallProfile`) is a named bundle of per-channel responses (Block / Overlap / Ignore) you assign to a component. Defined once in config, picked from a dropdown per mesh. *Like a named policy/ruleset you attach to objects rather than hand-coding pairwise checks.* See CONVENTIONS §4.
- **Online Subsystem (OSS) & NetDriver** — OSS is UE's pluggable backend abstraction (Steam/EOS/Null) for identity, sessions, friends; the *NetDriver* is the transport for game replication traffic. We route the game NetDriver through `SteamNetDriver` (dev AppId 480 = Valve's public test app) so the listen-server is reachable via Steam relay, with `IpNetDriver` as fallback. *OSS ≈ a provider interface with swappable implementations chosen by config.*
- **Enhanced Input (IMC / IA)** — UE5's input system. An *Input Action* (`IA_*`) is an abstract intent ("drag shot"); an *Input Mapping Context* (`IMC_*`) binds physical keys to actions and is pushed/popped per gameplay state with priorities. Both are editor `.uasset`s; code binds callbacks to the IAs. *Actions ≈ named commands; the mapping context ≈ a swappable keybinding profile layered at runtime.* Bind in C++ with `UEnhancedInputComponent::BindAction(Action, ETriggerEvent::Started/Completed/..., this, &Handler)`; add the context on possession via `UEnhancedInputLocalPlayerSubsystem::AddMappingContext`.
- **Rigid-body physics on a component** — a `UPrimitiveComponent` (we use a `USphereComponent`) becomes physical with `SetSimulatePhysics(true)`. *CCD* (continuous collision detection, `SetUseCCD`) stops fast objects tunnelling through thin walls. *Linear/Angular damping* bleed off velocity/spin (our roll feel). Forces: `AddImpulse(v, …, bVelChange)` — `bVelChange=false` respects mass (heavier ball = less launch); `AddForce(a, …, bAccelChange)` — `bAccelChange=true` applies a mass-independent acceleration (our custom roll-drag + boost). *No CLR analogue — this is a physics solver, not managed code.*
- **Console variables & commands (CVars)** — `TAutoConsoleVariable<float>`/`FAutoConsoleCommandWithWorldAndArgs` register `pb.*` knobs/commands reachable from the in-game console (backtick `` ` ``). Our pattern: a CVar defaults to `-1` meaning "use the authored UPROPERTY value"; any value `≥0` overrides it live. *Like runtime feature flags / `IConfiguration` you can poke without a redeploy — the backbone of feel-tuning (CONVENTIONS §1).*
- **WorldSubsystem** — a `UWorldSubsystem` is an engine-managed singleton scoped to a `UWorld`, auto-created/destroyed with the level, fetched via `World->GetSubsystem<T>()`. We use them for the camera director and the surface override stack. *A DI-container singleton with a lifetime tied to the level rather than the app.*
- **PrimaryDataAsset & Asset Manager** — `UPrimaryDataAsset` is a `UDataAsset` the engine's Asset Manager can discover/async-load by a typed id (`GetPrimaryAssetId` → `FPrimaryAssetType` + name). Our surface/powerup/hole definitions are primary assets so they can be enumerated and cooked deliberately. *A registered, discoverable config object vs. a plain POCO.*
- **Physical Material** — `UPhysicalMaterial` carries surface physics (friction, restitution) and is resolved from a trace's hit when `bReturnPhysicalMaterial` is set. Our `UPBPhysicalMaterial` also points at a `UPBSurfaceDefinition` so a hit resolves both the Chaos numbers *and* gameplay data in one lookup.
- **Automation Spec test** — `IMPLEMENT_SIMPLE_AUTOMATION_TEST(Class, "PuttBattle.Tests.…", flags)` registers a test the editor's Session Frontend (and `-ExecCmds="Automation RunTests …"`) can run. Guard pure-logic tests with `#if WITH_DEV_AUTOMATION_TESTS`. *NUnit/xUnit for the engine; runs in-editor, not in a packaged game.*
- **KillZ / FellOutOfWorld** — `WorldSettings.KillZ` is a world-space floor; an actor falling below it gets `AActor::FellOutOfWorld` (default: destroy). We override it on the ball to route to the checkpoint respawn instead (D7), so "fell off the map" and "fell in water" share one path.
- **Module include paths (Build.cs)** — UBT only auto-adds a module's `Public`/`Private` folders to the compiler include path. Our flat one-folder-per-subsystem layout has neither, so `PublicIncludePaths.Add(ModuleDirectory)` puts the module root on the path and lets headers be included by sub-path (`"Ball/PBBallPawn.h"`). *Like adding a source root to the compiler's `-I` set / a project's include dirs.*
